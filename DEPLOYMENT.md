# TableFlow AI - Production Deployment Guide

This guide outlines the steps required to take the TableFlow AI CRM from local development to a live production environment connected to real AI voice/chat agents.

## 1. Firebase Cloud Setup (Database & Auth)

Currently, the app relies on the local Firebase Emulators. To go live, you need a real Firebase project.

1. Go to the [Firebase Console](https://console.firebase.google.com/) and create a new project (e.g., `tableflow-production`).
2. Enable **Firestore Database** (start in production mode).
3. Enable **Authentication** (enable Email/Password sign-in).
4. Register a "Web App" in your Firebase project settings to get your production config object.

### Updating Environment Variables
In your `.env.local` (and your hosting provider's dashboard later), disable the emulators and add your real credentials:
```env
NEXT_PUBLIC_USE_FIREBASE_EMULATOR=false
NEXT_PUBLIC_FIREBASE_API_KEY="AIzaSy..."
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN="tableflow-production.firebaseapp.com"
NEXT_PUBLIC_FIREBASE_PROJECT_ID="tableflow-production"
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET="tableflow-production.appspot.com"
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID="..."
NEXT_PUBLIC_FIREBASE_APP_ID="..."
```

### Firebase Admin SDK (Crucial for AI Tool Calling)
Our AI API routes (`/api/ai/*`) safely bypass standard rules by using the server-side Admin SDK.
1. In the Firebase Console, go to **Project Settings > Service Accounts**.
2. Click **Generate New Private Key**. This downloads a JSON file.
3. Open the JSON file and extract the following into your `.env.local` and Vercel/Hosting env variables:
```env
FIREBASE_CLIENT_EMAIL="firebase-adminsdk-xxx@tableflow-production.iam.gserviceaccount.com"
FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\nMIIE... \n-----END PRIVATE KEY-----\n"
```
*(Note: If using Vercel, format the private key string exactly as it appears with the `\n` newlines, or follow Vercel's guide on multi-line secrets).*

### Deploying Firestore Rules & Indexes
To secure the database and make the UI queries fast, deploy your local configuration to the live cloud:
```bash
firebase deploy --only firestore:rules
firebase deploy --only firestore:indexes
```

## 2. Deploying the Next.js App

We recommend hosting the Next.js app on [Vercel](https://vercel.com) for zero-config global edge deployments.

1. Push this codebase to a GitHub repository.
2. Go to Vercel, select **Add New Project**, and import your repository.
3. Under **Environment Variables**, paste ALL the `NEXT_PUBLIC_...` and `FIREBASE_...` keys from Step 1.
4. Click **Deploy**. Vercel will build the app and give you a live URL (e.g., `https://tableflow-ai.vercel.app`).

## 3. Integrating with AI Voice/Chat Agents

Now that your backend is on the internet, you can hook it up to platforms like **Bland AI**, **Vapi.ai**, **Retell**, or **Voiceflow**.

### Adding "Tools" to your AI Agent
In your AI provider's dashboard, configure the following "Tools" (or Custom Functions) that the AI can call mid-conversation based on passenger intent:

#### 1. Check Availability
- **URL**: `POST https://yourdomain.com/api/ai/availability`
- **Description**: Call this to calculate open timeslots on a specific date.
- **Payload required by AI**: `tenant_id`, `date` (YYYY-MM-DD), `party_size`.

#### 2. Book Table
- **URL**: `POST https://yourdomain.com/api/ai/book`
- **Description**: Secures a new reservation.
- **Payload required by AI**: `tenant_id`, `date`, `time`, `party_size`, `customer_name`, `customer_phone`, `customer_email` (optional), and crucially an `idempotency_key` (e.g., UUID generated by the AI platform to prevent double-booking).

#### 3. Modify/Cancel Booking
- **URL**: `PUT https://yourdomain.com/api/ai/modify` or `DELETE https://yourdomain.com/api/ai/cancel`
- **Payload required by AI**: `tenant_id`, `reservation_id`, plus any fields changing (time, party_size), and `reason` for cancellation.

#### 4. Add to Waitlist
- **URL**: `POST https://yourdomain.com/api/ai/waitlist`
- **Description**: Add guest when no slots are available.
- **Payload required by AI**: Same as booking, plus `quoted_time_mins`.

### Setting up the Conversation Webhook (Post-Call)
Once the phone call ends, the AI needs to send the transcript to TableFlow so managers can view it in the "Conversations" Inbox.

1. Find the **"End of Call Webhook"** or **"Webhook URL"** in your AI dashboard settings.
2. Point it to: `POST https://yourdomain.com/api/webhooks/incoming-message`.
3. Read the payload structure in your provider's docs. You may need to adjust `src/app/api/webhooks/incoming-message/route.ts` to map the exact JSON structure your specific AI provider sends (e.g., mapping Bland AI's `transcript` array or Vapi's `summary` string to the `Conversation` type).

---

## 4. Final Security Check

- **HMAC Signatures**: Before accepting real restaurant clients, navigate to `/api/webhooks/incoming-message/route.ts` and add authentication (like verifying an HMAC signature header or a secret token) to ensure no one can spam your webhook endpoint.
- **Agent IDs**: Always ensure the AI platform sends a valid `agent_id` in requests so the `audit_logs` can properly trace which AI bot executed the action.

You are now ready to pilot TableFlow AI in a live environment!
